## 对照项目 / 基准

| 对照对象 | 相同点 | 不同点 | 对当前用户的含义 |
| --- | --- | --- | --- |
| 通用 LLM coding agent | 都会生成代码并执行。 | RD-Agent 把 hypothesis、experiment、feedback、trace 作为一等对象。 | 值得学习流程对象设计。 |
| Kaggle/MLE agent | 都面向 data science benchmark。 | RD-Agent 同时覆盖 quant 和多 R&D scenario。 | 不能只按 Kaggle agent 理解。 |
| 自研 AI alpha research system | 都可能需要 hypothesis、experiment、validation、memory。 | 自研系统需要更强金融安全边界和 validation discipline。 | RD-Agent 可启发，不宜直接替代。 |
| QuantGPT | 本轮不引用旧 notes。 | 需后续重新读取对照材料后再填正式表。 | 避免旧结论污染本轮。 |

## 对用户系统的启发

- 【建议】把 `Hypothesis`、`Experiment`、`Feedback`、`Decision`、`Trace` 显式建模。
- 【建议】每一轮都生成可审计 workspace 和 run id。
- 【建议】运行前必须有实验账本，运行后必须有 result/decision/next action。
- 【建议】LLM 和外部数据层需要人工审批和脱敏。
- 【建议】validation gate 必须由系统强制，而不是只写在 prompt 里。

## 仍需追加验证的问题

1. `fin_factor` 或 `fin_quant` 的真实输出里包含哪些指标。
2. Qlib backtest 是否强制使用 test/OOS，是否支持 rolling。
3. feedback decision 的标准是硬编码、prompt、还是 scenario evaluator。
4. session/UI 是否足够支持审计和复现。
5. CoSTEER knowledge management 是否保存失败模式并避免重复试错。
