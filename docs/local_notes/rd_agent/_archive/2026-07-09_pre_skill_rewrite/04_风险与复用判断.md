# RD-Agent 风险与复用判断

## 安全与隐私风险

【事实】RD-Agent 真实运行可涉及 LLM API、embedding API、Docker、Qlib 数据、Kaggle、Hugging Face、Azure/OpenAI/DeepSeek/LiteLLM provider。

【风险】可能发送到外部的内容包括：

- prompt。
- 研究问题和任务描述。
- 生成代码或错误日志。
- 数据集描述。
- financial report / paper 内容。
- quant factor / model idea。

【建议】涉及 alpha、策略、因子、代码、日志或 private data 时，不应默认上传给任何 provider。

## 账号、付费、云端、上传或交易风险

| 风险 | 说明 | 当前处理 |
| --- | --- | --- |
| OpenAI / Azure / DeepSeek / LiteLLM | chat 和 embedding 可能付费 | 未配置，未调用 |
| Docker | 可能拉镜像并执行生成代码 | 未运行 |
| Qlib 数据 | 可能下载或依赖本地数据版本 | 未运行 |
| Kaggle | 需要账号/token/接受比赛规则 | 未登录 |
| Hugging Face | 可能需要 token、license、模型下载、GPU | 未登录 |
| 云端 demo | 外部网站，结果不可当本地复现 | 仅作为官方资料 |
| 交易/真实提交 | 当前未见交易接口作为主线，但 quant 输出有策略风险 | 默认禁止 |

## 工程成熟度

【事实】`pyproject.toml` 标注 Development Status 为 `3 - Alpha`。

【事实】项目有 CI、tests、ruff、mypy、coverage 配置、docs 和 PyPI 包入口。

【推断】工程上不是一次性脚本，但也不能因为 Microsoft + paper + demo 就称为 production-ready。

【事实】本轮最小无外部调用 CLI 在未安装依赖时失败，说明直接源码 checkout 不是可用体验。

## 值得学习的部分

【建议】对我们的 AI alpha research system，RD-Agent 最值得学习：

1. hypothesis / task / code / feedback 的对象拆分。
2. R&D loop 的 trace 设计。
3. 用 Qlib 作为 quant execution backend 的思路。
4. 用 UI 展示 loop、代码和 feedback 的方式。
5. 将不同场景封装成 app/scenario 的结构。

## 可以直接复用的部分

【建议】当前阶段不建议直接复制源码模块。

可以先复用的只是设计概念：

- 研究过程 trace schema。
- “先体验 workflow，再读架构”的学习方式。
- Qlib 作为 backtest/evaluation layer 的候选。
- `loop_n` / `all_duration` 这类停止条件设计。

## 需要重写或隔离的部分

【建议】如果未来自研系统借鉴 RD-Agent，应重写或隔离：

- 外部 API 调用层：必须有审批、预算和数据发送审计。
- memory 层：必须显式保存 hypothesis、failure、decision、stop reason。
- validation 层：必须强制 OOS / rolling / cross-universe / sensitivity，而不是只看单次 test/backtest。
- CLI 轻量检查命令：应避免 import 重场景依赖。
- UI 审计层：必须让用户看到实验台账、变更、失败归因和下一步理由。

## 不适合当前系统的部分

【建议】当前不要照搬：

- Kaggle/FT-Agent 这种账号、数据、GPU、外部评测重依赖流程。
- 默认需要 LLM API 和 embedding API 的 agent loop。
- 未验证的 benchmark claim。
- 未验证的自动 hypothesis refinement。
- 将 README demo result 当成系统能力证据。

## 和已研究项目的区别

| 维度 | QuantGPT | RD-Agent |
| --- | --- | --- |
| 第一印象 | 本地 A 股 expression 回测器 + LLM/MCP 工具箱 | 通用 R&D agent framework |
| quant 执行 | 自己实现较多本地回测和评分 | 更多依赖 Qlib |
| workflow 叙事 | expression mining / tool workflow | hypothesis -> task -> code -> feedback |
| memory 风险 | task memory 强，research memory 弱 | trace vocabulary 更完整，但 durable memory 待验证 |
| 外部风险 | Cloud/WQ/DeepSeek/rqdatac | LLM/embedding/Docker/Qlib/Kaggle/HF/Azure 等更多 |

## 下一步建议

1. 【建议】先不跑 full demo，先决定是否允许安装依赖。
2. 【建议】若安装依赖，先验证 `rdagent health_check --no-check-env --no-check-docker`。
3. 【建议】若进入 quant 体验，写台账后只跑 `rdagent fin_factor --loop-n 1`，并记录 trace 是否真的包含 hypothesis、tasks、codes、feedback、metric 和 hypothesis_feedback。
