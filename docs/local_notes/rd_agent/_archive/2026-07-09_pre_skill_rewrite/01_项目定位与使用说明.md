# RD-Agent 项目定位与使用说明

## 一句话定位

【事实】RD-Agent 是一个面向 data-driven R&D 的 LLM agent framework，目标是让 AI 自动提出研究想法、实现代码、运行实验并根据反馈迭代。

## 作者意图

【事实】官方文档把 RD-Agent 描述为由两个核心部分组成：

- `Research`：提出新的 ideas / hypotheses。
- `Development`：把这些 ideas 实现成可运行代码。

【事实】README 和 scenario docs 展示的 happy path 不是“用户写一个函数然后调用库”，而是“用户选择一个场景，RD-Agent 自动进入 R&D loop”。

【推断】作者希望用户把 RD-Agent 当作自动研发 agent，而不是普通 Python 工具库。

## 目标用户与使用场景

【事实】官方展示的主要场景包括：

| 场景 | 命令 | 说明 |
| --- | --- | --- |
| Finance factor evolution | `rdagent fin_factor` | 自动提出和实现量化因子，并用 Qlib 回测 |
| Finance model evolution | `rdagent fin_model` | 自动提出和实现量化模型，并用 Qlib 评估 |
| Finance quant joint loop | `rdagent fin_quant` | 因子 + 模型联合优化，即 RD-Agent(Q) |
| Finance report factor extraction | `rdagent fin_factor_report` | 从金融研报提取因子并实现 |
| General model copilot | `rdagent general_model` | 从论文或报告实现模型 |
| Data Science / Kaggle | `rdagent data_science` | 自动特征工程、模型调参、比赛验证 |
| LLM fine-tuning | `rdagent llm_finetune` | 自动 fine-tuning 数据处理、训练和评测 |
| Log UI | `rdagent ui` / `rdagent server_ui` | 查看运行日志和 trace |

## 能做什么

【事实】从官方文档看，RD-Agent 可以：

- 用 LLM 生成 hypothesis / task / code。
- 在不同场景下运行实验。
- 在 quant 场景里接入 Qlib 做 factor/model/backtest。
- 通过 log/UI 查看 hypothesis、tasks、codes、feedback、metrics 等 trace。
- 在 data science / Kaggle / fine-tuning 场景里做更通用的模型研发工作。

## 不能做什么 / 不应这样用

【建议】当前阶段不应把 RD-Agent 当成：

- 已验证可盈利的 quant alpha 系统。
- 可以直接替代 QuantGPT 的本地表达式回测器。
- 无成本、无外部调用的本地工具。
- 可在 macOS 本地无准备跑通的 quickstart。
- 可以无审批调用 API、上传代码/数据、登录 Kaggle 或下载模型的系统。

【待验证假设】RD-Agent 是否有强制 research memory、failure memory、rolling validation、cross-universe validation，目前不能下结论。

## 最小使用路径

### 路径 A：只学习，不运行

【建议】当前最安全路径：

```text
读 README
读 docs/scens/catalog.rst
读 docs/scens/quant_agent_fin.rst
读 docs/installation_and_configuration.rst
读本地 notes
```

不需要 `.env`，不需要 API key，不需要 Docker。

### 路径 B：安装后做最小安全检查

【建议】如果用户批准安装依赖：

```bash
conda create -n rdagent python=3.10
conda activate rdagent
pip install rdagent
rdagent health_check --no-check-env --no-check-docker
```

注意：

```text
安装会联网下载依赖。
这个 health_check 只检查端口，不检查 API，不拉 Docker。
```

### 路径 C：最小 quant loop

【建议】如果用户批准 LLM API、Docker/conda、Qlib 数据和运行成本：

```bash
rdagent fin_factor --loop-n 1
```

第一次不要直接跑 `fin_quant`，因为它是因子 + 模型联合优化，范围更大。

## 依赖、账号、API key 和外部服务

| 项目 | 是否需要 | 风险 |
| --- | --- | --- |
| Python 3.10/3.11 | 推荐 | 当前本地 Python 3.12.9，不是官方主测版本 |
| Linux | 官方要求 | 当前是 macOS，不适合默认跑完整 demo |
| Docker | 多数场景需要 | 可能拉镜像、占磁盘、执行生成代码 |
| LLM chat API | 多数 agent loop 需要 | 可能付费，prompt/代码/研究上下文会发给 provider |
| embedding API | 多数检索/agent 能力需要 | 可能付费，检索内容会发给 provider |
| Qlib 数据 | quant 场景需要 | 数据下载、版本、PIT 对齐需要确认 |
| Kaggle token | Kaggle 场景需要 | 账号、比赛规则、API token |
| Hugging Face token | FT-Agent 可能需要 | 模型/数据 license、下载和 GPU 成本 |
| Azure/cloud | 可选 | 明确付费和上传边界 |

## 推荐第一次体验方式

【建议】第一次不要追求完整结果，先验证“体验链条”：

1. 确认环境和依赖。
2. 做无 API、无 Docker 的 `health_check`。
3. 准备 `.env`，但不把真实 key 写入 notes。
4. 写实验台账。
5. 只跑一轮 `fin_factor --loop-n 1`。
6. 看 trace 是否出现 hypothesis、tasks、codes、feedback、metric、hypothesis_feedback。
7. 再判断是否值得进入源码和复用研究。
