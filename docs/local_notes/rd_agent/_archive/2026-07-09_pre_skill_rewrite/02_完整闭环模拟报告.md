## 实际体验

【事实】即使关闭 env/API 和 Docker 检查，直接从源码运行 CLI 仍会先 import 多个 scenario module。当前未安装依赖时，命令在进入 health check 前就失败。

【推断】真实用户不能把源码 checkout 当成可直接运行的脚本目录；至少要先走 `pip install rdagent` 或 `make dev`。

【推断】RD-Agent 的 CLI 有一定“重入口”特征：一个轻量命令也会受到其它场景依赖影响。这对我们设计自研系统有提醒价值：低风险检查命令最好保持极少依赖。

## 失败、绕路和修正

【事实】失败：

```text
ModuleNotFoundError: No module named 'fire'
```

【事实】`requirements.txt` 中包含 `fire`。

【建议】修正路径不是单独安装 `fire`，而是按官方安装整个依赖环境：

```bash
conda create -n rdagent python=3.10
conda activate rdagent
pip install rdagent
```

或开发模式：

```bash
make dev
```

但安装会联网下载依赖，需用户确认。

## 外部调用、成本和安全边界

【事实】本轮没有外部 API 调用，没有 Docker 拉镜像，没有 Qlib 数据下载，没有账号登录。

【事实】如果进入真实 `fin_factor` / `fin_quant`，会至少涉及 LLM API 和 Qlib/Docker/数据环境。

【建议】不要把 `.env.example` 复制成真实 `.env` 后直接运行；需要先确认 provider、预算、发送内容和日志保存位置。

## 和 README / docs 承诺的差异

【事实】README 的 quickstart 假设用户已经安装 RD-Agent 和依赖，并且在 Linux/Docker/LLM API 配置可用的环境里运行。

【事实】本地当前源码 checkout + Python 3.12.9 不能直接运行最小 CLI 命令。

【推断】README 更像“已准备好环境后的使用说明”，不是“任意本地 checkout 后可直接运行”的教程。

## 结论

【建议】下一次真实体验前，先决定是否允许联网安装依赖。安装后先跑最小端口检查，再考虑 LLM/API 测试。

【建议】如果要体验 quant 场景，先选 `fin_factor --loop-n 1`，不要直接上 `fin_quant`。
