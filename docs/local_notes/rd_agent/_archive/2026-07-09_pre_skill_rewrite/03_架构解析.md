# RD-Agent 架构解析

## 一句话架构

【推断】RD-Agent 的架构可以先理解为：

```text
CLI / app 选择场景
-> scenario 定义任务语境
-> Research 生成 hypothesis / tasks
-> Development / coder 生成并执行代码
-> evaluator / feedback 收集结果
-> log / UI 展示 trace
```

当前只做项目级架构，不深入每个 package 和 class。

## 核心模块

| 模块 | 作用 | 当前证据 |
| --- | --- | --- |
| `rdagent/app/` | CLI 和各应用入口 | `rdagent/app/cli.py` 暴露多个命令 |
| `rdagent/core/` | 核心抽象框架 | 官方 development docs 说明 core 是抽象框架 |
| `rdagent/components/` | 可复用组件 | 官方 docs 说明 components 被 scenarios 复用 |
| `rdagent/scenarios/` | 具体场景 | 包含 Qlib、data science、Kaggle、finetune 等 |
| `rdagent/log/` | 日志、UI、trace server | `rdagent/log/server/README.md` 给出 message schema |
| `rdagent/oai/` | LLM backend | README / config docs 涉及 OpenAI、Azure、DeepSeek、LiteLLM |
| `web/` | 新 Web UI | README 说明可配合 `server_ui` 实时查看 |
| `docs/` | 官方文档 | Sphinx docs |

## 主要 workflow / data flow

### Finance factor workflow

【事实】官方 docs 对 `fin_factor` 的描述是：

```text
hypothesis generation
-> factor creation
-> factor implementation
-> Qlib backtesting
-> feedback analysis
-> hypothesis refinement
```

【推断】这是最适合作为第一次 quant demo 的路径，因为它比 `fin_quant` 范围更窄。

### Finance model workflow

【事实】官方 docs 对 `fin_model` 的描述是：

```text
hypothesis generation
-> model creation
-> model implementation
-> Qlib backtesting
-> feedback analysis
-> hypothesis refinement
```

### Finance quant workflow

【事实】官方 docs 把 `fin_quant` 描述为 RD-Agent(Q)，强调 factor-model co-optimization。

【推断】它更接近作者想展示的完整 quant R&D loop，但也更重、更难控制成本。

## 关键文件阅读顺序

【建议】后续若进入源码阶段，不应从全仓库逐文件读起，而应围绕运行路径读：

1. `rdagent/app/cli.py`：命令入口。
2. `rdagent/app/qlib_rd_loop/factor.py`：`fin_factor` 入口。
3. `rdagent/app/qlib_rd_loop/model.py`：`fin_model` 入口。
4. `rdagent/app/qlib_rd_loop/quant.py`：`fin_quant` 入口。
5. `rdagent/scenarios/qlib/`：Qlib 场景对象、proposal、experiment。
6. `rdagent/log/`：trace 和 UI。
7. `rdagent/components/coder/`：生成代码和执行反馈。

## 状态、日志、memory 或持久化

【事实】`rdagent/log/server/README.md` 定义了 trace message schema。

一轮 loop 中可能出现：

- `research.hypothesis`
- `research.tasks`
- `evolving.codes`
- `evolving.feedbacks`
- `feedback.config`
- `feedback.return_chart`
- `feedback.metric`
- `feedback.hypothesis_feedback`

【推断】RD-Agent 的 auditability 至少在 trace vocabulary 上比 QuantGPT 的普通 task history 更接近“研究过程记录”。

【待验证假设】这些 trace 是否完整、是否持久、是否能跨 session 检索失败经验，还需要真实运行或读 log storage 代码确认。

## 测试、验证和质量控制

【事实】`pyproject.toml` 定义了 `offline` pytest marker，说明项目区分“不需要外部 API 的测试”。

【事实】README 和 scenario docs 展示 Qlib train/valid/test 和 backtest 逻辑。

【待验证假设】quant 场景是否有强制 rolling validation、cross-universe validation、ablation、anti-overfit discipline，目前没有验证。

## 如果是 AI / agent / quant / backtest 项目

| 层 | 当前理解 |
| --- | --- |
| Research Agent | 【事实】官方 docs 强调生成 hypothesis / idea |
| Main Agent / Orchestrator | 【推断】由 app/scenario loop 组织 |
| Reviewer Agent | 【待验证假设】是否有独立 reviewer 或只是 feedback evaluator 未确认 |
| Backtest / Simulation Layer | 【事实】quant 场景使用 Qlib |
| Data Layer | 【事实】Qlib 示例使用本地 `~/.qlib/qlib_data/cn_data` |
| Memory / Findings / Failures | 【待验证假设】docs 提到 knowledge / feedback，但失败记忆强度未确认 |
| Experiment Tracking | 【事实】trace schema 有 loop_id、evo_id、metric、feedback |
| Human Approval Gate | 【待验证假设】官方 quickstart 没看到强审批机制；我们自己的 protocol 必须补 |
| Anti-overfit / Validation Discipline | 【待验证假设】固定 train/valid/test 已见，rolling/cross-universe 未确认 |
| Report / UI / Explanation Layer | 【事实】有 Streamlit UI 和新 Web UI |
